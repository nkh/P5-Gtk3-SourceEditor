# Proposal: Replace `:bindings` TextView with Gtk3::TreeView

**Project:** P5-Gtk3-SourceEditor v0.04
**File affected:** `lib/Gtk3/SourceEditor/VimBindings/Command.pm`
**Author:** nkh
**Date:** 2026-04-05

---

## 1. Background and Motivation

The `:bindings` ex-command currently opens a modal dialog that dumps all keybindings into a read-only `Gtk3::TextView`. The text is formatted as a monospace, 3-column layout generated by `generate_bindings_text()` (Command.pm lines 345–530) and rendered as a single flat string inside a `Gtk3::TextView` wrapped in a `Gtk3::ScrolledWindow`.

### Current limitations

| Issue | Detail |
|---|---|
| **No interactivity** | The user sees a wall of text. There is no way to collapse/expand mode sections, search within bindings, or click on an entry. |
| **Fragile column layout** | The 3-column format is built with `sprintf` and fixed-width paddings (10-char keys, 20-char descriptions). It breaks on long descriptions or narrow windows. |
| **No sorting or filtering** | All ~100+ bindings across 6 modes are shown at once with no way to narrow the view. |
| **Poor visual hierarchy** | Mode sections are separated only by ASCII heading lines (`---`). There is no visual distinction between the key column and the description column — it is all plain monospace text. |
| **No theme awareness** | The bindings dialog does not use the editor's active theme colors. It renders with default system colors, making it visually disconnected from the editor. |
| **No keyboard navigation inside the dialog** | The TreeView widget natively supports arrow-key navigation and type-ahead search, which would allow the user to find a binding without closing the dialog and switching back to the editor. |

### Why Gtk3::TreeView

`Gtk3::TreeView` is the standard GTK3 widget for displaying structured, tabular data. It provides built-in support for:

- **Sortable columns** — click a column header to sort ascending/descending.
- **Type-ahead search** — start typing to jump to matching rows.
- **Expandable tree nodes** — mode sections can be collapsed/expanded.
- **Row selection** — single or browse selection for highlighting a binding.
- **Column resizing** — drag column borders to adjust width.
- **Fixed headers** — column headers remain visible while scrolling.
- **Theme integration** — renders using the current GTK theme, respecting the editor's `ThemeManager` colors.

## 2. Current Implementation

The relevant code lives in two functions in `Command.pm`:

### `_show_bindings_dialog()` (lines 290–339)

```perl
sub _show_bindings_dialog {
    my ($ctx) = @_;
    my $tv = $ctx->{gtk_view} or return;
    my $text = generate_bindings_text($ctx);

    my $parent = eval { $tv->get_toplevel } // undef;
    my $window = Gtk3::Window->new('toplevel');
    $window->set_title('Bindings');
    $window->set_transient_for($parent) if $parent;
    $window->set_default_size(900, 520);
    $window->set_modal(TRUE);

    my $scroll = Gtk3::ScrolledWindow->new();
    $scroll->set_policy('automatic', 'automatic');
    $scroll->set_border_width(6);

    my $textview = Gtk3::TextView->new();
    $textview->set_editable(FALSE);
    $textview->set_wrap_mode('none');
    $textview->set_cursor_visible(FALSE);
    # ... margins, monospace font ...
    $buf->set_text($text);

    $scroll->add($textview);
    # ... close button, vbox, show_all ...
}
```

### `generate_bindings_text()` (lines 345–530)

This function collects bindings from `$ctx->{resolved_keymap}` and `$ctx->{ex_cmds}`, translates GDK key names to human-readable strings via `%key_name`, maps action names to descriptions via `%desc`, deduplicates visual-mode keys already shown in normal mode, and formats everything into a 3-column text layout:

```
-- NORMAL MODE --
------------------
h          move right           j           move down
k          move up              l           move right
...
-- INSERT MODE --
------------------
Escape     back to normal
...
```

The function returns a plain string. It is also used by the test suite (headless), so it must remain available for testing.

### Data structures consumed

| Source | Structure | Example keys |
|---|---|---|
| `$ctx->{resolved_keymap}{normal}` | `{ key => action_name, _ctrl => { key => action }, _char_actions => { key => action } }` | `h`, `j`, `dd`, `x` |
| `$ctx->{resolved_keymap}{insert}` | Same shape | `Escape`, `BackSpace` |
| `$ctx->{resolved_keymap}{replace}` | Same shape | `Escape`, `BackSpace` |
| `$ctx->{resolved_keymap}{visual}` | Same shape | `y`, `d`, `c`, `>>`, `~` |
| `$ctx->{resolved_keymap}{command}` | Same shape | `Escape` |
| `$ctx->{ex_cmds}` | `{ command_name => action_name }` | `w`, `q`, `wq`, `e`, `s` |
| `%key_name` (local) | `{ gdk_name => display_name }` | `dollar => '$'` |
| `%desc` (local) | `{ action_name => description }` | `delete_char => 'delete char'` |

## 3. Proposed Design

### 3.1 Widget hierarchy

```
Gtk3::Window (toplevel, modal, transient)
  └── Gtk3::Box (vertical)
       ├── Gtk3::SearchEntry (optional, top bar — see §3.4)
       └── Gtk3::ScrolledWindow
            └── Gtk3::TreeView
                 └── Gtk3::TreeStore (3 columns: Mode, Key, Action)
```

The existing Close button is dropped in favor of the native window close button (the `X` in the title bar) and the `Escape` key, which is standard GTK dialog behavior. If a button bar is desired for consistency with the existing `:browse` dialog, a single Close button can be retained at the bottom.

### 3.2 TreeStore model

The `Gtk3::TreeStore` will use **two levels**:

- **Level 0 (parent rows):** One row per mode section — "Normal", "Insert", "Replace", "Visual", "Command", "Ex Commands". These rows are expandable/collapsible. The Key and Action columns are empty for parent rows.
- **Level 1 (child rows):** Individual key bindings with populated Key and Action columns.

| Column | Type | Content (parent) | Content (child) |
|---|---|---|---|
| **Mode** | `Glib::String` | Section heading (e.g. "NORMAL MODE") | Empty string |
| **Key** | `Glib::String` | Empty | Human-readable key (e.g. `dd`, `Ctrl-d`, `:w`) |
| **Action** | `Glib::String` | Empty | Description (e.g. "delete line (dd)") |

Column types in `Gtk3::TreeStore->new('Glib::String', 'Glib::String', 'Glib::String')`.

### 3.3 TreeView columns

| Column ID | Title | Sortable | Resizable | Width (px) |
|---|---|---|---|---|
| 0 | "Mode" | No (tree structure) | No | auto |
| 1 | "Key" | Yes | Yes | 160 |
| 2 | "Action" | Yes | Yes | fill remaining |

The "Mode" column functions as the tree expander. It is not sortable because the tree structure defines the visual order. The "Key" and "Action" columns are sortable — clicking a header sorts the entire tree (or, optionally, only the children within each mode section using a `Gtk3::TreeModelSort` with a custom sort function).

#### Cell renderers

- **Mode column:** `Gtk3::CellRendererText`, bold weight, `markup` property for the section heading (e.g. `<b>— NORMAL MODE —</b>`).
- **Key column:** `Gtk3::CellRendererText`, monospace font family, left-aligned. This makes keys like `dd`, `Ctrl-d`, and `:w` visually distinct from descriptions.
- **Action column:** `Gtk3::CellRendererText`, default (proportional) font, left-aligned.

### 3.4 Optional: SearchEntry for filtering

A `Gtk3::SearchEntry` at the top of the dialog allows the user to filter bindings by typing. This is a progressive-enhancement feature that can be added in a follow-up after the basic TreeView is working. When the user types in the search entry, a `Gtk3::TreeModelFilter` wraps the `TreeStore` and hides rows whose Key or Action columns do not match the search string (case-insensitive substring match). Parent rows are shown as long as they have at least one visible child.

If this is included in the initial implementation, the widget hierarchy becomes:

```
Gtk3::Window
  └── Gtk3::Box (vertical)
       ├── Gtk3::Box (horizontal, search bar)
       │    ├── Gtk3::Label ("Filter:")
       │    └── Gtk3::SearchEntry
       ├── Gtk3::ScrolledWindow
       │    └── Gtk3::TreeView
       └── Gtk3::ButtonBox (horizontal)
            └── Gtk3::Button ("Close")
```

### 3.5 Theme integration

The existing `ThemeManager` returns `fg` and `bg` color values extracted from the active XML theme. The current code applies these to the mode label and command entry via CSS. The same approach extends to the bindings dialog:

```perl
my $theme = $ctx->{theme};  # { fg => '#...', bg => '#...', css_provider => ... }
```

Two options for applying the theme:

**Option A — CSS provider (preferred):** Add CSS rules for the TreeView widget to the existing `$ui_css` string in `ThemeManager::load()`:

```css
GtkTreeView#bindings_tree {
    background-color: <bg>;
    color: <fg>;
}
GtkTreeView#bindings_tree header button {
    background-color: <bg>;
    color: <fg>;
}
```

This requires passing the CSS provider into `_show_bindings_dialog()`, either via `$ctx` or as a new parameter.

**Option B — Inline on TreeView:** Set the `Gtk3::CssProvider` directly on the TreeView widget using `Gtk3::Widget::get_style_context->add_provider()`. This keeps the change self-contained in `Command.pm` and does not require modifying `ThemeManager.pm`.

Option B is recommended for the initial implementation to minimize the diff surface. Option A can be adopted later for full theme consistency.

## 4. Implementation Plan

### Phase 1: Core TreeView replacement

**Scope:** Replace the `Gtk3::TextView` with `Gtk3::TreeView` using a `Gtk3::TreeStore`. No search, no theme changes.

**Steps:**

1. **Write `generate_bindings_list()`** — a new function in `Command.pm` that returns the same data as `generate_bindings_text()` but as a structured Perl array instead of a formatted string:

   ```perl
   # Returns: [
   #   { mode => 'NORMAL MODE', bindings => [
   #       { key => 'h', action => 'move left' },
   #       { key => 'Ctrl-d', action => 'half page down' },
   #       ...
   #   ]},
   #   ...
   # ]
   sub generate_bindings_list { ... }
   ```

   This reuses the existing `%key_name`, `%desc`, `$build_from_keys`, `$build_ctrl`, and `$build_char_actions` closures from `generate_bindings_text()`. Extract those shared helpers into private subs (e.g. `_build_from_keys($km)`, `_build_ctrl($km)`, etc.) and call them from both `generate_bindings_text()` (for the test suite) and `generate_bindings_list()` (for the TreeView).

2. **Rewrite `_show_bindings_dialog()`** — replace the `Gtk3::TextView` construction with `Gtk3::TreeStore` + `Gtk3::TreeView`:

   ```perl
   sub _show_bindings_dialog {
       my ($ctx) = @_;
       my $tv = $ctx->{gtk_view} or return;
       my $parent = eval { $tv->get_toplevel } // undef;

       my $window = Gtk3::Window->new('toplevel');
       $window->set_title('Key Bindings');
       $window->set_transient_for($parent) if $parent;
       $window->set_default_size(700, 500);
       $window->set_modal(TRUE);

       # TreeStore: Mode | Key | Action
       my $store = Gtk3::TreeStore->new('Glib::String', 'Glib::String', 'Glib::String');

       my $sections = generate_bindings_list($ctx);
       for my $section (@$sections) {
           my $mode_iter = $store->append(undef);
           $store->set($mode_iter, 0, $section->{mode}, 1, '', 2, '');
           for my $b (@{$section->{bindings}}) {
               my $child_iter = $store->append($mode_iter);
               $store->set($child_iter, 0, '', 1, $b->{key}, 2, $b->{action});
           }
       }

       my $treeview = Gtk3::TreeView->new_with_model($store);
       $treeview->set_headers_visible(TRUE);
       $treeview->set_enable_search(TRUE);
       $treeview->expand_all;

       # Column 0: Mode (expander)
       my $renderer_mode = Gtk3::CellRendererText->new();
       $renderer_mode->set('weight', 700);  # bold
       my $col_mode = Gtk3::TreeViewColumn->new_with_attributes(
           'Mode', $renderer_mode, text => 0
       );

       # Column 1: Key (monospace)
       my $renderer_key = Gtk3::CellRendererText->new();
       my $font_desc = Pango::FontDescription->from_string('Monospace 10');
       $renderer_key->set('font-desc', $font_desc);
       my $col_key = Gtk3::TreeViewColumn->new_with_attributes(
           'Key', $renderer_key, text => 1
       );
       $col_key->set_sort_column_id(1);
       $col_key->set_resizable(TRUE);
       $col_key->set_min_width(120);

       # Column 2: Action
       my $renderer_action = Gtk3::CellRendererText->new();
       my $col_action = Gtk3::TreeViewColumn->new_with_attributes(
           'Action', $renderer_action, text => 2
       );
       $col_action->set_sort_column_id(2);
       $col_action->set_resizable(TRUE);
       $col_action->set_expand(TRUE);

       $treeview->append_column($col_mode);
       $treeview->append_column($col_key);
       $treeview->append_column($col_action);

       # Escape to close
       $window->signal_connect(key_press_event => sub {
           my ($w, $event) = @_;
           if ($event->keyval == Gtk3::Gdk::keyval_from_name('Escape')) {
               $w->destroy;
               return TRUE;
           }
           return FALSE;
       });

       my $scroll = Gtk3::ScrolledWindow->new();
       $scroll->set_policy('automatic', 'automatic');
       $scroll->set_border_width(6);
       $scroll->add($treeview);

       $window->add($scroll);
       $window->show_all;
   }
   ```

3. **Preserve `generate_bindings_text()`** — keep the existing function unchanged so that the 210+ headless tests continue to pass. The two functions share the extracted helper closures, so there is no code duplication.

4. **Test manually** — run the editor, type `:bindings`, verify the TreeView renders with all 6 mode sections, expand/collapse works, column sorting works, and Escape closes the dialog.

### Phase 2: Theme integration

**Scope:** Apply the editor's active theme colors to the TreeView.

**Steps:**

1. **Pass theme data via `$ctx`** — ensure `$ctx->{theme}` is populated with `{ fg, bg, css_provider }` when the editor loads a theme. Check whether `SourceEditor.pm` already stores this in the context; if not, add it.

2. **Apply CSS to TreeView** — in `_show_bindings_dialog()`, after creating the `$treeview`:

   ```perl
   if (my $theme = $ctx->{theme}) {
       my $css = Gtk3::CssProvider->new();
       my $css_str = qq{
           GtkTreeView { background-color: $theme->{bg}; color: $theme->{fg}; }
           GtkTreeView header button { background-color: $theme->{bg}; color: $theme->{fg}; }
       };
       eval {
           $css->load_from_data(Encode::encode('UTF-8', $css_str));
           $treeview->get_style_context->add_provider($css, 600);
       };
       warn "bindings theme error: $@" if $@;
   }
   ```

3. **Verify** with each bundled theme (default, dark, light, solarized).

### Phase 3 (optional): SearchEntry filtering

**Scope:** Add a search/filter bar above the TreeView.

**Steps:**

1. **Add `Gtk3::SearchEntry`** to the top of the dialog's VBox.
2. **Wrap the TreeStore in a `Gtk3::TreeModelFilter`** — connect the `visible` function to check each child row's Key and Action columns against the search text.
3. **Auto-expand sections with matches** — when filtering, expand parent rows that have visible children; collapse rows with no matches.
4. **Clear on Escape** — if the SearchEntry is focused, Escape clears the filter text rather than closing the dialog. A second Escape press closes the dialog.

## 6. Risks and Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| `Gtk3::TreeStore` API differences across GTK3 versions | Low | The TreeStore API has been stable since GTK 3.0. The Perl bindings (`Gtk3`) wrap the C API directly with no abstraction changes. |
| Sort on child rows breaks tree grouping | Medium | Use a `Gtk3::TreeModelSort` wrapper and override the default sort to sort within each mode section, or disable sort and rely on the natural insertion order (which is already alphabetical from `generate_bindings_text()`). |
| Performance with 100+ rows | Very Low | TreeView handles thousands of rows efficiently. 100–150 rows is trivial. |
| Test suite regression | None | `generate_bindings_text()` is preserved unchanged. All 210+ tests pass without modification. |
| Theme CSS not applied on some platforms | Low | Fall back to system defaults. The `eval {}` block around CSS loading prevents crashes. |
